# UI Redesign

React 페이지의 UI를 수정·재디자인한다.
비즈니스 로직(API, 상태, 라우팅)은 100% 보존하면서 시각적 부분만 교체한다.

---

## 스킬 시작 — 세션 초기화 (1회만)

**⚠️ STOP: 아래 3가지 중 하나를 사용자에게 반드시 물어본 뒤 진행한다. 사용자가 선택하기 전까지 어떤 수정도 시작하지 않는다.**
**모드는 세션 시작 시 1회만 선택한다. 이후 루프에서는 다시 묻지 않는다.**
**단, 사용자가 명시적으로 "모드 바꿔줘"라고 하면 재선택 허용.**

```
어떤 방식으로 수정할까요?

A. URL 비교 수정 — 참조 URL의 레이아웃을 현재 페이지에 맞추기
B. 디자인 시안 비교 수정 — 이미지/HTML 파일 기준으로 맞추기
C. 자유 수정 — DESIGN-SYSTEM.md 기반으로 지시에 따라 수정
```

### A. URL 비교 수정 — 초기 정보 수집
- 사용자에게 **참조 URL**(목표 디자인)과 **수정 대상 URL**(변경할 페이지)을 물어본다
- page-capture로 참조 URL 스크린샷을 캡쳐한다 (**세션 내 1회만 캡쳐, 이후 루프에서 재사용**)
- 목적: 이미 완성된 페이지를 참조하여 다른 페이지를 비슷한 톤으로 맞추기

### B. 디자인 시안 비교 수정 — 초기 정보 수집
- "참조 폴더 경로를 알려주세요. (기본값: `.person/pic/`)" 질문 추가
- 지정 폴더를 스캔하여 이미지(`.png`, `.jpg` 등) / `.html` 파일 목록 확인
  - **파일이 없으면**: "지정 폴더에 참조 파일이 없습니다. 다른 경로를 입력하거나 C 모드(자유 수정)로 전환할까요?"
  - 사용자 선택: 경로 재입력 / C 모드 전환
- 사용자에게 **수정 대상 URL**(변경할 페이지)을 물어본다
- 목적: 외부 디자인 시안(Stitch, Figma 캡쳐 등)을 기존 페이지에 적용

### C. 자유 수정 — 초기 정보 수집
- "어떤 페이지를 수정하시겠습니까?" 라고 물어본다
- 목적: 매번 디자인 규칙을 말하지 않고 자동 적용

### 공통 초기화
- **DESIGN-SYSTEM.md를 반드시 읽는다** (`.claude/rules/design-system.md`) — **세션 1회만**
- `current_target`: 현재 수정 중인 대상 파일 경로 (루프 내 유지)

## 성역 (절대 수정 금지)

| 성역 | 이유 |
|------|------|
| 상단 헤더 (AppLayout Header) | 전역 레이아웃 — 좌: 사이드바 트리거 + 브레드크럼, 우: 기관명 > 과정명 > 상태 배지 |
| 사이드바 (Sidebar) | 전역 내비게이션 |
| API 호출 로직 (useQuery, fetch 등) | 백엔드 연동 |
| 라우팅 (useNavigate, Link) | 페이지 이동 |
| 상태 관리 (useState, useContext) | 비즈니스 로직 |
| 조건부 렌더링 로직 (삼항 연산자 등) | 비즈니스 로직 및 상태에 따른 UI 제어 |
| 브레드크럼 (Breadcrumb) | 페이지 이동 + 사용자 경험 |
| 기능 컴포넌트 (마크다운 에디터, 파일 업로드, 서명 캔버스 등) | 기존 기능 보존 |

기존 페이지의 기능 컴포넌트의 디자인 변경은 가능하지만, 변경된 디자인에 그 기능이 다 들어가야한다.
수정 후에도 기존 비즈니스 로직은 100% 동작해야 한다.

검증 방법:
- hook 호출(`useQuery`, `useState`, `useNavigate` 등)이 삭제되지 않았는지 확인
- 조건부 렌더링의 조건식이 변경되지 않았는지 확인
- navigate 경로, API 엔드포인트가 변경되지 않았는지 확인

## [루프] 수정 실행 — "그만" 전까지 반복

> **루프 구조 요약:**
> - 수정 완료 후 "✅ 수정 완료! 다음 지시를 입력하세요." 메시지 출력
> - 다음 사용자 입력을 받아 루프 계속 또는 종료
> - 모드(A/B/C)는 루프 전체에서 유지

### Step 1. 사전 분석 (동일 대상 반복이면 skip)

**skip 조건**: `current_target`이 이전 루프와 동일한 경우 → Step 2로 바로 이동
**반드시 실행**: 첫 번째 루프이거나 대상 페이지가 변경된 경우

0. 대상 페이지 코드를 읽고 성역(API, 상태, 로직)을 식별한다
1. 같은 타입(LIST, FORM, DETAIL)의 기존 페이지 2~3개를 읽어 패턴을 파악한다
2. (A) page-capture로 수정 대상 URL의 현재 스크린샷을 캡쳐하여 참조 캡쳐와 비교한다
3. (B) 폴더 내 이미지 파일을 읽고, 수정 대상 URL의 현재 스크린샷을 캡쳐하여 비교한다. `.html` 파일이 있으면 레이아웃 구조도 참조한다

### Step 2. 디자인 매핑
- (A) 참조 URL의 레이아웃·스타일 분석. **DESIGN-SYSTEM.md 색상·컴포넌트 규칙 우선** 적용, 참조 URL과 충돌 시 사용자에게 확인
- (B) 이미지 + `.html` 파일에서 레이아웃 구조 추출. **DESIGN-SYSTEM.md 색상·컴포넌트 규칙 우선** 적용, 시안과 충돌 시 사용자에게 확인
- (C) 사용자 지시 + DESIGN-SYSTEM.md + 기존 패턴 기반으로 수정
- DESIGN-SYSTEM.md 표준을 기본으로 따르되, 새로운 디자인이 기존 표준과 충돌하면 사용자에게 질문한다:
  - "기존 디자인 시스템에 맞춰 비슷하게 적용할까요, 아니면 새로운 디자인으로 변경할까요?"

### Step 2-5. 더미 데이터 규칙 (더미 데이터 사용 시 항상 적용)
더미 데이터를 추가하거나 수정할 때 — 신규 페이지든 기존 페이지 수정이든 — 반드시 다음 절차를 따른다.
임의 필드(백엔드에 없는 필드)는 어떠한 상황에서도 사용 금지.

1. **백엔드 스키마 먼저 확인**: `backend/src/main/java/.../dto/` 또는 `entity/`에서 해당 도메인의 실제 필드를 확인한다
2. **프론트엔드 타입도 확인**: `frontend/src/types/` 에서 대응하는 TypeScript 타입의 필드명·Enum값을 확인한다
3. **더미 인터페이스를 스키마 기준으로 정의**: 백엔드 DTO의 필드명과 동일하게 작성한다
4. **주석 표기**: 각 필드 옆에 `// 대응하는 백엔드 필드명` 주석을 남긴다
5. **API 연결 위치 명시**: 더미 배열 위에 `// TODO: API 연결 시 이 블록을 교체` 주석을 추가한다

### Step 3. 코드 수정
- 성역 보존하면서 JSX 구조 재구성
- **모든 UI 텍스트는 한국어**
- 아이콘은 프로젝트 표준 아이콘으로 대체
- 페이지 내부에 별도 `<header>` 생성 금지
- **⚠️ AppLayout 내 최상위 wrapper 절대 규칙**: AppLayout 안의 페이지는 이미 레이아웃 패딩이 적용되어 있다.
  - 최상위 wrapper는 반드시 `<div className="space-y-4">` 또는 `<div className="space-y-6">` 만 사용
  - `flex`, `min-h-screen`, `items-start`, `justify-center`, `bg-background`, `px-*`, `py-*`, `max-w-*` **절대 추가 금지**
  - 잘못된 예: `<div className="flex min-h-screen items-start justify-center bg-background px-4 py-10"><div className="w-full max-w-2xl">`
  - 올바른 예: `<div className="space-y-4">`
  - 참조 페이지(SubjectFormPage, CourseFormPage 등)가 `space-y-*` 하나만 쓰는지 반드시 확인 후 동일하게 적용
- **화면 적합 밀도 원칙**: 가능하면 주요 콘텐츠가 스크롤 없이 한 화면에 보이도록 설계한다.
  - 여백/패딩을 과도하게 크게 잡지 않는다 (`p-8` 이상은 지양, `p-5~6` 권장)
  - 카드 내 수치 폰트는 `text-2xl`을 최대치로 (`text-3xl` 이상 지양)
  - 차트/그래프 높이는 `h-[180px]~h-[220px]` 범위 권장 (`h-[300px]` 이상 지양)
  - 요약 통계 카드는 `p-5`, 리스트 아이템은 `py-3~4` 권장
  - 단, 폼 페이지(FORM)는 입력 편의를 위해 이 원칙을 완화 적용
- **표준 헤더 보장**: 페이지에 `useBreadcrumb()` 훅이 없으면 반드시 추가한다.
  브레드크럼 구성: `[{label: "상위메뉴", path: "/..."}, {label: "현재페이지"}]`
  AppLayout 헤더(사이드바 트리거 + 브레드크럼 + 과정 정보)는 절대 제거·수정 금지
- **⚠️ Markdown Viewer/Editor**: 페이지에 MD 뷰어·에디터가 있으면 **반드시** `.claude/reference/md-patterns.md`를 읽고 해당 패턴을 그대로 적용한다.
  - 4가지 패턴(Viewer Dark · Viewer Light · Editor Dark · Editor Light) 중 맥락에 맞는 것을 선택
  - Editor 패턴에는 Notion 가져오기 버튼 + ◧⊞◨ 뷰 전환이 반드시 포함되어야 함
  - 패턴 파일을 읽지 않고 임의로 구현하는 것을 금지
  - 패턴 파일 경로: `.claude/reference/md-patterns.md`

### Step 4. 검증 (tsc)
- `tsc` 빌드 오류 확인
- diff에서 성역(hook, 조건부 렌더링, navigate) 삭제 여부 확인
- 기능 컴포넌트 유실 여부 확인
- **tsc 오류 발생 시:**
  1. 자동으로 오류 수정 시도 (1회)
  2. 수정 후 tsc 재실행
  3. 여전히 오류 시 → 사용자에게 오류 내용 보고 후 지시 대기 (루프 중단 없음)

### Step 5. 완료 메시지 + 다음 지시 대기 (루프 핵심)

수정 1회 완료 후:
```
✅ 수정 완료! 다음 지시를 입력하세요. ('그만' 입력 시 종료)
```

**사용자 입력 처리:**

| 입력 유형 | 처리 |
|-----------|------|
| 종료 키워드 (단독) | 루프 종료 → rule-update 발동 (새 패턴이 있는 경우) |
| 다른 페이지 지시 | "새로운 대상 페이지를 수정하겠습니다." → `current_target` 갱신 → Step 1부터 |
| 동일 페이지 추가 수정 | Step 3부터 (Step 1 skip) |
| "모드 바꿔줘" | 모드 재선택 → 현재 루프 계속 |

**종료 키워드 목록:** `그만`, `stop`, `종료`, `완료`, `done`, `끝`

**⚠️ 종료 키워드 오인식 방지 규칙:**
- 종료 키워드가 **단독 메시지**이거나 "수정 그만", "그만 해줘" 같은 짧은 종료 의사 표현인 경우에만 종료 처리
- 문장 안에 포함된 경우 (`"stop 애니메이션 제거"`, `"종료 버튼 추가해줘"`)는 **절대 종료로 처리하지 않는다**
- 판단 기준: 메시지 전체가 종료 의사 표현인지 여부

---

## 루프 종료 처리

종료 키워드 감지 시:
1. 세션 동안 새로운 UI 패턴(색상·컴포넌트·여백·레이아웃)이 추가되었다면 → `rule-update` 스킬 발동
2. 변경된 패턴이 없으면 → "작업을 마칩니다." 로 종료

## 전체 플로우 요약

```
스킬 시작
  └─ 모드 선택 (A/B/C) — 1회
  └─ 모드별 초기 정보 수집
       A: 참조 URL 캡쳐 (1회) + 수정 대상 URL
       B: 폴더 경로 + 파일 유무 확인 + 수정 대상 URL
       C: 수정 대상 페이지
  └─ DESIGN-SYSTEM.md 읽기 — 1회

[루프 시작] ← current_target 유지
  Step 1. 사전 분석 (동일 대상 반복이면 skip)
  Step 2. 디자인 매핑
  Step 3. 코드 수정
  Step 4. 검증 (tsc)
           └─ 실패 시: 자동 수정 1회 → 여전히 실패 시 보고 후 대기
  Step 5. 완료 메시지 + 다음 지시 대기
    ├─ 종료 키워드 (단독) → [루프 종료] → rule-update (필요 시)
    ├─ 다른 페이지 지시 → current_target 갱신, Step 1부터
    └─ 동일 페이지 추가 수정 → Step 3부터 (Step 1 skip)
```
