## 목적

실수가 있었다면 → 원인을 분석하고 재발을 막는다.
구현이 완료됐다면 → 새로 생긴 규칙·패턴을 rules에 반영한다.
두 경우 모두 rules 파일을 업데이트하고 changelog에 기록한다.

---

## 진입 경로 판단

| 상황 | 진입 |
|------|------|
| 실수·버그·오동작 발생 | **A. 실수 분석** → Step 0부터 |
| 구현 완료 후 규칙 갱신 | **B. 규칙 갱신** → Step 2부터 |

---

## A. 실수 분석 (why)

### Step 0. 문제 확인

**인자 없이 발동된 경우** (예: `/why` 단독):
> 어떤 문제가 있었나요? 간단히 설명해주세요.

→ 사용자 답변을 받은 뒤 Step 1 진행

**인자와 함께 발동된 경우** (예: `/why 레거시 페이지 제거 시 ...`):
→ 인자를 문제 설명으로 사용, Step 0 skip, Step 1 바로 진행

### Step 1. 원인 분류

| 유형 | 설명 |
|------|------|
| **규칙 누락** | 해당 케이스를 다루는 룰이 없었음 |
| **규칙 모호** | 룰이 있지만 해석이 달라질 수 있었음 |
| **잘못된 가정** | 코드/도메인에 대해 틀린 전제를 깔았음 |
| **컨텍스트 미확인** | 관련 파일을 읽지 않고 작업함 |
| **지시 오독** | 사용자 지시를 다르게 해석함 |

### Step 1-1. 원인 보고

```
**원인 유형**: {유형}
**무슨 일이**: {실수 1줄 요약}
**왜 틀렸나**: {근본 원인 설명}
**앞으로는**: {추가할 규칙 초안}
```

→ 이후 Step 2(규칙 추가) 진행

---

## B. 규칙 갱신 (rule-update)

세션 대화를 분석하여 사용자가 명시적으로 지시한 내용을 분류한다:

- **코드 규칙**: 코딩 컨벤션, 파일 구조, 네이밍, 설정 관련 지시
- **디자인 규칙**: UI/UX 패턴, 색상, 컴포넌트 사용법 지시
- **비즈니스 규칙**: 역할, 권한, 도메인 로직, 상태 흐름 변경

→ Step 2(규칙 추가) 진행

---

## Step 2. 갱신 대상 파일 선택

| 실수·변경 성격 | 파일 |
|--------------|------|
| 코드 작성 방식, 파일 구조, 확인 순서 | `code-rules.md` |
| UI 컴포넌트, 색상, 레이아웃 | `design-system.md` |
| 도메인 로직, 권한, 상태 흐름 | `business-rules.md` |

---

## Step 3. 규칙 작성 위치 판단 (rules/ vs reference/)

**rules/에 직접 쓴다:**
- 규칙·제약·금지사항·선택 기준 (코드 없이 문장으로 표현 가능)
- 20줄 이하이고 코드 블록 없음

**reference/로 추출한다:**
- 코드 블록이 포함된 구현 패턴
- 20줄 초과 섹션
- 절차:
  1. `.claude/reference/{도메인}-patterns.md` 생성
  2. rules/ 해당 섹션을 요약 + 포인터로 교체: `> 상세 패턴: .claude/reference/{파일명}.md`
  3. `.claude/reference/README.md` 표에 행 추가

**규칙 문체:**
- 금지: "~하지 않는다 / ~금지"
- 의무: "반드시 ~ 먼저 확인한다 / ~해야 한다"

---

## Step 4. Changelog 기록

`.claude/logs/{파일명}-changelog.md`에 행을 prepend한다:

| 날짜 | 변경 항목 | 이전 | 이후 | 이유 |
|------|----------|------|------|------|
| `오늘날짜` | {실수 요약 또는 갱신 항목} | 이전 규칙 | 추가·변경된 규칙 요약 | {원인 유형 또는 구현 완료} |

- 파일 없으면 헤더 포함 신규 생성
- rules 파일 자체에는 Changelog 테이블을 넣지 않는다
- rules 파일 최하단에 포인터가 없으면 한 번만 추가: `> 변경 이력: .claude/logs/{파일명}-changelog.md`
- 최대 20건 유지 (초과 시 오래된 행 제거)

---

## Step 5. 완료 보고

```
✓ 원인: {유형} (실수 분석 경로인 경우)
✓ 갱신 파일: {파일명} > {섹션}
✓ 위치: rules/ 직접 추가 | reference/ 추출
✓ Changelog: .claude/logs/{파일명}-changelog.md
```
